# Auth Service (auth_api)

## Общее описание

**Цель сервиса:** Обеспечение централизованной аутентификации, авторизации и управления пользователями и их ролями в системе.

**Стек технологий:**
- **Язык:** Python 3.14+
- **Фреймворк:** FastAPI
- **База данных:** PostgreSQL (SQLAlchemy + Alembic)
- **Кэширование/Сессии:** Redis

**Зависимости:**
- Внешние API: Не использует внешние сторонние API напрямую.
- Интеграции: Является базовым сервисом идентификации, который предоставляют токены доступа для других микросервисов системы (например, `movies_api`).

---

## Бизнес-логика и правила

### Ключевые сущности
- **User**: Пользователь системы. Содержит учетные данные (`email`, хеш пароля) и статус активности.
- **Role**: Роль пользователя (например, `admin`, `subscriber`). Определяет уровень доступа к ресурсам. Разделяются на системные (неизменяемые) и пользовательские.
- **TokenVersion**: Версионность токенов для каждого пользователя. Позволяет мгновенно отозвать все активные сессии пользователя.
- **UserAuthHistory**: Лог успешных входов в систему с фиксацией времени и User-Agent устройства.

### Основные бизнес-правила
- **Безопасность паролей:** Пароли хранятся исключительно в виде хешей (используется `argon2` или `bcrypt`).
- **Управление ролями:** 
    - Системную роль (`is_system=True`) запрещено изменять или удалять.
    - Названия ролей приводятся к нижнему регистру для исключения дублей при разном регистре ввода.
- **Жизненный цикл сессии (JWT):** 
    - Используется пара `Access Token` и `Refresh Token`, передаваемые через HTTP-only cookies.
    - При обновлении токена (`refresh`) старая пара токенов заносится в черный список (Blacklist) для предотвращения повторного использования.

### Жизненный цикл
1. **Login** $\rightarrow$ Создается запись в `UserAuthHistory` $\rightarrow$ Генерируется версия токена $\rightarrow$ Выдаются Access и Refresh токены.
2. **Access Token Expired** $\rightarrow$ Запрос с Refresh-токеном $\rightarrow$ Проверка версии и черного списка $\rightarrow$ Старые токены в Blacklist $\rightarrow$ Выдача новой пары токенов.
3. **Logout (Single)** $\rightarrow$ Текущие токены заносятся в черный список Redis $\rightarrow$ Удаление cookies.
4. **Logout All Devices** $\rightarrow$ Инкремент `TokenVersion` пользователя $\rightarrow$ Все существующие JWT с предыдущей версией становятся невалидными.

### Интеграции
- **Входящие:** Вызывается фронтендом или API Gateway для регистрации, входа и управления профилем/ролями.
- **Исходящие:** Взаимодействует с Redis (кэширование версий токенов) и PostgreSQL (хранение данных).
